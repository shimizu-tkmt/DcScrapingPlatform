# データベース構成提案書 (ローカル & 本番)

ローカルデバッグの利便性と、本番環境の安全性・スケーラビリティを両立するための構成を提案します。

---

## 1. 構成の概要

環境に応じて接続先とデータベースエンジンを切り替える構成を推奨します。

| 環境 | 推奨構成 | 理由 |
| :--- | :--- | :--- |
| **開発 (Local)** | **SQLite** または **Neon 開発ブランチ** | 軽量さ重視なら SQLite、本番環境(Postgres)との互換性重視なら Neon ブランチ。 |
| **本番 (Prod)** | **Neon (Main Branch)** | サーバーレス PostgreSQL による低コスト・高可用性の維持。 |

---

## 2. 詳細な実装案

### 案A：SQLite による究極の軽量構成（推奨）
ローカルに PostgreSQL をインストールすることなく、ファイル一つで DB を完結させます。

- **実装**: `Program.cs` で `builder.Environment.IsDevelopment()` に応じて `UseSqlite()` と `UseNpgsql()` を切り替える。
- **メリット**: セットアップ不要。起動が速い。
- **デメリット**: 一部の Postgres 固有機能が使えない可能性がある（本プロジェクトの現状のモデル範囲なら問題なし）。

### 案B：Neon ブランチによる高精度構成
Neon の「Branching」機能を使い、開発用のクローン DB に対してデバッグを行います。

- **メリット**: 本番と全く同じ機能・型が使える。
- **デメリット**: ネットワーク環境（インターネット接続）が必要。

---

## 3. 設定管理のベストプラクティス

接続文字列の「ハードコード（直書き）」を排除し、安全に管理します。

1.  **User Secrets (ローカル開発用)**:
    - `dotnet user-secrets` を使用。ソースコード（Git）に接続文字列を含めない。
2.  **appsettings.Development.json (共有開発設定用)**:
    - チーム内で共有可能な非機密設定を保持。
3.  **環境変数 (本番用)**:
    - 実行環境（Azure/Render等）の Secret 機能で管理。

---

## 4. 推奨されるアクション

1.  **`appsettings.json` から接続文字列を削除**: 機密情報の漏洩を防ぐ。
2.  **`appsettings.Development.json` の導入**: ローカル用の設定（SQLite 等）を定義。
3.  **`Program.cs` の修正**: 環境に応じた DB プロバイダーの切り替え。
